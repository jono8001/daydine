# V4 Operator Report — Samples Assessment

*Authored April 2026. Branch: `claude/verify-v4-scoring-spec-L4Cjv`.*
*Scope: Stratford trial. Samples under `samples/v4/monthly/`. Engine `v4.0.0`.*

## Purpose

A structured read of the seven sample V4 operator reports generated by
`scripts/generate_v4_samples.py`. Confirms whether the redesigned V4
shell behaves correctly under each evidence / rankability state, whether
the previously strong V3.4 commercial sections still feel commercially
strong after remapping, and what should be refined before any broader
rollout.

All 24 guardrail tests pass and all seven samples emit with **0 errors
and 0 warnings** from the QA layer (§10 of the V4 Report Spec).

## 4. Awkward wording and structural issues found

### 4.1 Low-value evidence rows in Market Position

In Rankable-A / Rankable-B samples the Market Position table renders as:

```
| Peer ring | Position | Peer avg | Peer top |
|---|---|---:|---:|
| Local (5 mi) | — | — | — |
| Catchment (15 mi) | — | — | — |
| UK category cohort | — | — | — |
```

This is honest — the sample runner passes `peer_benchmarks=None` so the
helper fills em-dashes. Correct by spec, but the reader gets a three-row
empty-looking table. A production wiring should either populate this
table via `peer_benchmarking.compute_benchmarks` or suppress the table
body entirely when no ring has data, leaving a short explainer. A
minimal fix is to render one line — "peer benchmarks not yet computed
for this month" — in place of an empty table.

### 4.2 Synthetic temp-closed sample leaks `league_table_eligible = Yes`

The Opposition sample header reads:

```
**Confidence class:** Rankable-B
**Rankable:** Yes  ·  **League-eligible:** Yes
```

A real temp-closed venue would have `league_table_eligible = False`
(spec §7.4 row 2) because the engine would detect the closure and
drop league eligibility. The sample script sets `business_status =
CLOSED_TEMPORARILY` on the venue record but does **not** re-run the
scoring engine, so the V4 payload still reports `league = True`. The
temp-closure banner fires correctly at the top of the report, but the
inline Score / Confidence & Rankability Basis block contradicts it.

This is a sample-runner artefact, not a report-generator bug. Two
fixes are reasonable: (a) re-score the synthetic fixture through the
engine before rendering; (b) have the generator override
`league_table_eligible` to False when it sees a temporary-closure
state. Option (b) is safer because it catches real-world races where
the engine and the closure flag are out of sync.

### 4.3 Profile-only-D "How to unlock full scoring" direction wording

The Roebuck sample reads:

> Establish public customer-platform evidence. A single venue platform
> profile with at least five reviews moves this venue out of
> Directional-C into Rankable-B.

Roebuck is currently Profile-only-D, not Directional-C. The aspirational
step is D → Directional-C → Rankable-B. The phrasing should ladder
clearly: first D → C (one primary family + ≥4 signals), then C → B
(two families + ≥7 signals + ≥10 reviews). The unblock-action builder
in `v4_report_generator._directional_unblock_action()` is shared between
Directional-C and Profile-only-D paths, which produces the awkward
"out of Directional-C" phrasing for a D venue.

Fix: split the unblock builder into two — one for D (D → C is the next
step) and one for C (C → B is the next step).

### 4.4 Closed sample evidence appendix still shows pre-closure data

Arrow Mill (synthetic Closed) renders:

```
| Google rating | 4.5 |
| Google review count | 1988 |
| Website present | True |
```

These are last-observed values from before the closure synthesis. The
report is a closure notice; showing the pre-closure evidence helps
audit but can read as implying the venue is open. Suggest adding a
one-line header to the Evidence Appendix in Closed mode: *"Data below
reflects the last observed state before closure was detected; it does
not describe current operation."*

### 4.5 Directional-C Financial Impact renders full figures

Soma (Directional-C) renders Financial Impact as:

> **Confidence: Low.** Figures are directional.
> | Monthly revenue impact | — | £180 – £1,350 |
> **Recommended action cost band:** £200 – £1,000
> **Expected payback window:** 1 – 3 months
> > *Directional-C:* the headline score is not league-ranked. These
> > figures are illustrative only …

Spec §6.1 allows either a Low-confidence full rendering **or** the
canonical Directional-C fallback wording. The current generator always
picks the full rendering. The figures are guarded ("illustrative only
— not a target") but a reader skimming the section sees numbers first.
This is arguably the V3.4 failure mode persisting: front-and-centre
£ figures attached to a venue whose score is explicitly not
league-ranked.

Recommendation: make the Directional-C path fall back to the §6.4
canonical Directional-C wording by default, and surface the figures
only on an explicit override. Keeps the discipline airtight; operators
who want the sizing can still drill into the Commercial Readiness
section.

### 4.6 Soma / Vintner Financial Impact figures are identical

Both samples render £180 – £1,350 / month at stake. Same single gap
("no booking / contact path"), same `_MONTHLY_COVERS` fallback table,
same `gpl` default. The model does not yet reflect venue-level context
(Vintner has 887 Google reviews at £££; Soma has 120 Google reviews
at ££; they should not come out with identical leakage estimates).

Fix: include Customer Validation review volume and the CR component
score in the sizing math, not just the gap count. At minimum, apply a
coverage scaling factor keyed to review volume (log₁₀) so highly-
reviewed venues with thin CR show a larger absolute £ ceiling than
lightly-reviewed venues with the same gap.

### 4.7 `gpl` (price level) is still the primary Financial Impact input

`_fi_estimate` calls `commercial_estimates._spend_range(gpl)` and
`_MONTHLY_COVERS.get(gpl)` as the primary drivers. V4 Report Spec §6.5
forbids `gpl` **as the sole basis**. The current generator uses it
alongside a per-gap leakage multiplier (observed gap count × 2%),
which technically meets the spec's "sole basis" test, but `gpl` is
doing most of the work. This is the single item most likely to drift
back into an overclaim.

Fix: replace the `gpl`-keyed covers / spend tables with CR-component
shape (CR score → implied recoverable range) so price level is a
weak prior, not the anchor.

### 4.8 Profile-only-D has a double blank line after the title

Cosmetic. The header renderer inserts a blank line; the Profile-only-D
orchestrator does not emit the separator `---` that the full report
uses. Result is a visually thin head. Low priority.

### 4.9 Management Priorities / Watch List / Implementation Framework render empty

Every live V4 sample produces empty or near-empty content in these
sections because `recommendations=None` in the sample runner. Structure
is correct; content is pending pipeline wiring. Expected for a samples
run, but the assessment should note that a production report won't be
evaluable until the recommendations engine is plumbed through.

## 5. Sections that still feel too V3-ish

A focused read looking for residual V3.4 framing that slipped past the
re-seat:

### 5.1 Directional-C suppresses sections that spec says are "Conditional"

`SECTIONS_DIRECTIONAL_C` in `v4_report_spec.py` does not list:
- Watch List (spec §5.11: "C" for Directional-C — **conditional**, not suppressed)
- What Not to Do This Month (spec §5.12: "C")
- Next-Month Monitoring Plan (spec §5.15: "C")

The Directional-C orchestrator in the generator matches the spec list
and therefore skips them. This is a stricter-than-spec choice. For
Directional-C venues these sections are arguably the most useful — the
operator has an unblock path and should be watching something while
they work on it.

Fix: render Watch List, What Not to Do, and Next-Month Monitoring in
Directional-C mode with a reduced scope (monitoring anchored on the
unblock path, WNTD anchored on V4-specific no-lever items, Watch List
defaulting to the standard fallback). The guardrails will still block
any V3-era phrasing; adding the sections does not reintroduce risk.

### 5.2 `_dimension_to_component` mapping is a compatibility shim

The Management Priorities / Implementation Framework renderers map a
legacy `dimension` field on each recommendation to a V4 component
name:

```python
mapping = {
    "trust": "Trust & Compliance",
    "experience": "Customer Validation",
    "visibility": "Customer Validation",
    "conversion": "Commercial Readiness",
    "prestige": "Distinction",
}
```

This is necessary while the V3.4 recommendations engine is still
producing V3.4 dimension codes. Once the recs engine is refactored to
emit V4 component codes natively, the shim should be removed — keeping
it long-term risks drift when the recs engine changes shape. Flag as
a later cleanup.

### 5.3 `commercial_estimates` still references `gpl`-keyed tables

Noted in §4.7 above. The Financial Impact section calls V3.4 helpers
(`_spend_range`, `_MONTHLY_COVERS`) which are `gpl`-keyed. The helpers
are internally V3.4-shaped. The spec §6.5 line against `gpl` as a
primary driver means this shim is on the critical path for removal.

### 5.4 Demand Capture Audit builder still expects a V3.4-style scorecard

`_build_demand_capture_audit` constructs a V3.4 scorecard stub on the
fly and passes it to the existing `run_demand_capture_audit`. The
existing audit reads the stub only to colour its verdicts (it does not
affect the score). But the stub is a wart — it keeps the V3.4 scorecard
schema alive inside the V4 renderer. Later passes should give the
audit a V4-aware variant that consumes `ReportInputs` directly.

### 5.5 Segment / Menu / Review narrative sub-blocks are untouched V3.4

`segment_analysis.py`, `menu_intelligence.py`, `review_analysis.py`,
`review_delta.py` are reused verbatim. They produce correct narrative
output. They still contain internal V3.4 framing ("aspect scores", the
five-aspect taxonomy, hard-coded 30-day recent-movement window) that
is invisible to the V4 renderer because the narrative is delivered as
themes, not numbers. No user-facing issue; worth a cosmetic refactor
later so the taxonomies are configurable rather than coded.

### 5.6 No residual 5-dim scorecard or verbal-band language in any sample

Grepping across all seven `.md` outputs for the banned strings
(`Dimension Scorecard`, `Excellent band`, `Generally Satisfactory`,
`Major Improvement`, `Urgent Improvement`, `v3.3`, `v3.4 Premium`,
`prestige dimension`) returns **zero matches**. The V3 frame has been
cleanly replaced at the rendering layer.

## 6. Refinements before broader rollout

Priority-ordered, each with the effort estimate if obvious.

### 6.1 Must-fix before any operator sees a V4 report

1. **Directional-C: render Watch List, What Not to Do, and Monitoring
   Plan** per spec §5.11 / §5.12 / §5.15. Add them to
   `SECTIONS_DIRECTIONAL_C` and extend the `_render_directional_c_report`
   orchestrator. Small.
2. **Temp-closed: `league_table_eligible` must be False.** Either
   re-score the fixture, or have the generator override the flag when
   a temp-closure banner is rendered. Small.
3. **Profile-only-D unblock wording ladder.** Split
   `_directional_unblock_action` into `_d_unblock_action` (D → C) and
   `_c_unblock_action` (C → B). Small.
4. **Directional-C Financial Impact defaults to canonical fallback.**
   Only render figures on an explicit override. Removes the
   "illustrative only" wrapper as a primary operator-facing surface.
   Small.
5. **Closed Evidence Appendix needs a "last observed" header line.**
   One-line change. Small.

### 6.2 Should-fix before the report is commercially published

6. **Wire peer benchmarks into the sample runner** so Market Position
   renders non-empty tables. Use `peer_benchmarking.compute_benchmarks`
   with a Rankable-* class filter (add the filter if not present).
   Medium.
7. **Wire recommendations into the sample runner** so Management
   Priorities / Watch List / What Not to Do / Implementation Framework
   render real content. Requires a thin adapter that returns V4-ready
   recs from the existing recs engine. Medium.
8. **Replace `gpl`-keyed Financial Impact sizing with a
   CR-component-keyed model** so `gpl` is a weak prior rather than the
   anchor. Medium.
9. **Venue-level Financial Impact scaling by review volume.** Log-scale
   adjustment keyed to Customer Validation platform counts so highly-
   reviewed venues don't produce identical £ ceilings to thin-review
   venues. Small.
10. **Segment Intelligence: apply the class-demoted review tier** to
    segment commercial-read prose, not just to praise / criticism
    openers. Small, requires touching `segment_analysis` consumer path.

### 6.3 Nice-to-have post-rollout

11. **Remove `_dimension_to_component` shim** when the V3.4 recs engine
    is retired. Delete map; update recs engine to emit V4 codes natively.
12. **V4-aware demand-capture audit builder.** Drop the V3.4 scorecard
    stub.
13. **CI integration** (spec §10.5): wire
    `python -m tests.test_v4_report_guardrails` into the CI step so
    any regression is blocked at merge.
14. **Penalty explanation registry.** Move `penalty_explanation()`'s
    hard-coded table to a data file so new penalty codes added to the
    scoring engine auto-populate the "What this means" column.
15. **Tolerance check on Financial Impact ranges** (noted in the
    guardrails work): a "Moderate" confidence claim with a suspiciously
    narrow £ range should trip a warning. Low priority.

## 7. Readiness verdict

**The V4 report layer is structurally ready.**

Every class renders with the correct shape. Mandatory sections appear
where required; suppressed sections never leak. The guardrail test
suite catches regressions into V3-era overclaiming, and all seven
samples emit with 0 errors and 0 warnings. None of the samples produced
any of the banned phrasings or prohibited cross-class content.

It is **not content-complete** yet — the five Must-fix items in §6.1
are short but block a real operator-facing rollout. The Should-fix
items in §6.2 are where the actual commercial strength of the report
lives (peer benchmarks and recommendations must be populated for the
restored sections to feel as strong as they did in V3.4). Neither set
requires scoring-engine changes; both are wiring work on top of the
existing V3.4 helpers.

The report layer can be used internally today for diagnostic QA
against the scoring engine. It should not be used as a live operator
deliverable until §6.1 is complete, and it should not be
marketed / priced until §6.2 is complete.

## 9. Main lessons from the samples

1. **The mode branching works.** Five modes (Rankable-A, Rankable-B,
   Directional-C, Profile-only-D, Closed, plus Temp-closed as a
   Rankable variant) each render with a distinct, deliberate shape.
   This was the structural question; answered cleanly.
2. **The restored V3.4 sections survive intact.** Every section in the
   brief's eight-item list is present where the spec says it should
   be. Where content reads thin, the cause is missing wiring
   (benchmarks, recs), not missing structure.
3. **The guardrails catch what matters.** The 24-test suite blocks
   every risk from the tightening prompt (sentiment drives score,
   convergence uplift, price / photos / social / delivery as score
   driver, V3 dimension language, six verbal bands, Directional-C
   without caveat, Profile-only-D with score headline, Closed with
   zero score, entity ambiguity not surfaced, Financial Impact without
   confidence / cost / payback, ROI claims, precise score-movement
   promises). Sample runs confirm zero false positives.
4. **The failure mode that survives is Financial Impact sizing.**
   `gpl` (price level) doing most of the arithmetic is the single
   place where V3.4 framing leaks through a V4 discipline check.
   Identical £ ceilings for dissimilar venues is the diagnostic
   symptom. High-priority fix per §6.2.
5. **Directional-C is correctly gated, but slightly over-gated.**
   Watch List, What Not to Do, and Next-Month Monitoring should render
   for Directional-C per spec and they currently do not. Small gap;
   fix is mechanical (§6.1).
6. **Synthetic closure fixtures exposed one real generator gap** —
   the temp-closed sample leaks `league_table_eligible = True`.
   Round-tripping synthetic fixtures through the scoring engine or
   defensively overriding the flag in the renderer closes this.
7. **The compact decision trace + collapsible raw-trace pattern works.**
   Spec §9 calls the decision trace "explanatory, not a dump". The
   current rendering gives a one-line "Formed from: Trust X +
   Customer Y + Commercial Z. Final: N" summary, a penalty-with-
   explanation table, and a `<details>` wrapper around the raw engine
   trace. Operator-facing ergonomics are right.

Stopping.
